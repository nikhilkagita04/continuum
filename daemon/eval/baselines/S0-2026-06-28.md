# S0 baseline — retrieval & answer quality (first measured gate)

**Date:** 2026-06-28 · **Code:** `daemon/eval/measure.mjs` @ the commit that adds this file ·
**Verdict:** CONDITIONAL_GO (6-person distinguished panel, unanimous) — proceed to Sprint-A on the
*direction*; these numbers are **not** ratified as a passed gate (small N, polluted corpus).

> This record is **aggregate-only by design**: the probe questions and retrieved snippets contain the
> developer's private captured screen content and are kept **local** (never committed to this public
> repo). Reproduce on your own store with the command below.

## Question
Does the agent answer your *real* questions from captured doing, **better than the bare model**?

## Method (what the engine actually runs now)
- **Probes** = `{question, expected_answer}` pairs auto-generated by a frontier model from the user's
  own salient moments, then **validated deterministically** — the expected answer must actually appear
  in the source moment (`answerInSource`, numeric-aware: numbers must match exactly, the discriminating
  token is required). Avoids the unanswerable-probe trap.
- **Cross-family, no self-grading** (eval-plan §0.2): answer + baseline = Claude (deployed agent class);
  judge = a *different* family. **Gate-of-record = deterministic** ("does the answer contain the expected
  fact"), fully reproducible; the LLM judge is reported alongside as a cross-check.
- **Same retrieval path as production** — `scoreProbes` now applies `routeSearch` (recency routing),
  matching `mcp.mjs` / `retrieval.mjs`.
- **hit@5** = the exact source moment (by content-hash) was retrieved in top-5 (strict). The product-felt
  metric is **correctness** (the agent gives the right answer, possibly from another answer-bearing moment).

## Result (N=15, dogfooding store of 2,396 episodes; judge = local gemma3)
| Metric | Point | Wilson 95% CI | Gate | |
|---|---|---|---|---|
| hit@5 (exact source) | 0.33 | 0.15 – 0.58 | ≥0.70 | ❌ below |
| **correctness** (deterministic) | **0.47** | 0.25 – 0.70 | — | judge cross-check 0.53 |
| groundedness | 0.47 | 0.25 – 0.70 | →1.0 | |
| **necessity** (bare model could NOT) | **0.47** | 0.25 – 0.70 | ≥0.50 | ⚠️ straddles |
| latency | p50 184ms · p95 ~200–530ms | — | <150ms | ❌ bge-m3 query path |

## Honest reading
- **Every metric's CI straddles its gate** — at N=15 this is not a pass/fail, it is noise. Do not claim
  the promise is empirically met yet.
- **Retrieval is the single bottleneck.** Failures are retrieval misses, not answer failures; correctness
  (0.47) exceeds exact-source hit (0.33) precisely because the agent answers from answer-bearing moments
  that are not the one probe-source. Zero fabrication observed beyond not-answering.
- **The magic is real but unproven at scale:** verified private facts the bare model cannot know
  (e.g. a contact's employer; a named person's role at the company they work at — present multiple
  times in the store) were answered grounded.
- **Corpus is not the product's test:** ~41% of the store is the act of building Continuum itself
  (design docs, this planning conversation, the repo). External validity for "general knowledge-work
  captured" is low.

## Must-fix before this is a trustworthy gate (panel)
1. Re-gate on a **clean, leakage-controlled, adequately-powered** corpus (N≥40–50; paraphrase probe
   questions to kill BM25 lexical leakage; gold set curated blind to system output; report CI lower bound).
2. **Scrub + consent-gate the eval egress** — `measure.mjs` currently sends raw text to the judge/probe
   model with no `scrub`; make local judging the default and gate any cloud judge behind explicit consent.
   Reconcile `perception.md:55` / `capture-quality.md:97`.
3. Ship a **reranker over a widened candidate pool** (hybrid top-50 → cross-encoder → top-5) — the
   verified highest-leverage retrieval move — *before* the ANN/33M-embedder latency swap.

## Sprint-A retrieval: cross-encoder reranker — MEASURED, then REFUTED by the clean re-gate
First A/B (N=15, mixed/meta probe set) looked like a big win: answer-bearing hit@5 0.67 → 0.87, exact
0.33 → 0.60 (ms-marco-MiniLM-L-6-v2 ONNX, pool=20). **The clean re-gate overturned it.** On a blind,
**paraphrased**, external-activity-only gold set (N=34), the reranker shows **no benefit — it slightly
hurts**, and a controlled 2×2 on the same sources isolates why:

| answer-bearing hit@5 | baseline | + reranker |
|---|---|---|
| ORIGINAL question (leaky) | 0.79 | 0.71 |
| PARAPHRASED question (leakage-controlled) | 0.62 | 0.56 |

- **Paraphrasing drops baseline 0.79 → 0.62 (−0.17)** — that gap *was* BM25 lexical leakage (probes minted
  from source text reuse its tokens). The true clean baseline is ~0.62 answer-bearing / **0.38 exact**.
- **The reranker hurts in both conditions** (−0.08, −0.06; N=34, CIs ±~0.15 so "no benefit" is the safe
  claim). The off-the-shelf MS MARCO cross-encoder is **out-of-distribution on noisy screen-OCR** and
  mis-ranks. The earlier +0.20 was a probe-set + leakage artifact.

**Stronger models don't fix it** (answer-bearing hit@5 on the clean paraphrased gold, baseline 0.62):
ms-marco-MiniLM-L-6 (22M) → 0.56; bge-reranker-base (278M) → 0.56 (6× slower); bge-reranker-v2-m3 gated,
mxbai-rerank-v2 is a qwen2 decoder Transformers.js can't run, jina-v3 is non-commercial. A 12× bigger
model gives the *identical* negative result ⇒ it's not model strength, it's the corpus/first-stage.

**The bottleneck is first-stage retrieval, located by the recall curve (answer-bearing):**
`PARAPHRASED  r@5 0.62  r@10 0.71  r@20 0.76  r@50 0.82  r@100 0.91` — for hard queries the answer-bearing
moment is often *not retrieved at all* (r@100 only 0.91), which no reranker can fix.

**Decision: REVERTED.** Removed `@huggingface/transformers` + the reranker module/wiring — `continuum-core`
is zero-dependency again. The cross-encoder had no proven payoff on representative data. Real levers:
(a) a **clean, larger external-activity corpus** (this store yielded only 34 clean probes; r@100 0.91 means
some answers aren't findable), and (b) better **first-stage** retrieval (a stronger/faster query embedder
that fixes recall AND latency together). Add a model dependency only when one of those *measures* a win.

## S0 EXTERNAL-ONLY GATE — the representative read (clean, paraphrased, N=46)
Excluding Claude-app captures (the 985-episode Continuum-build conversation) + obvious build-meta leaves a
1,124-episode external corpus (LinkedIn/X/Chrome/WhatsApp) — a proxy for a normal laptop. Indexing AND
probing over external-only, blind + paraphrased, N=46 (clears the N≥40 power bar):

| Metric | Point | Wilson 95% CI |
|---|---|---|
| hit@5 (exact source) | 0.57 | 0.42 – 0.70 |
| **hit@5 (answer-bearing)** | **0.76** | — |
| correctness | 0.41 | 0.28 – 0.56 |
| necessity | 0.39 | 0.26 – 0.54 |
| latency p50/p95 | 165 / 171 ms | — |

**Retrieval is healthy once the build-noise is out of the index** — answer-bearing hit@5 jumps 0.33 → 0.76
vs the polluted store; the 985 build episodes were massive distractors. **The bottleneck has flipped to the
ANSWER LAYER:** the answer-bearing moment is retrieved 76% of the time but the agent surfaces the correct
fact only 41%. Decomposed (re-scored on the same gold set):
- **~13 pts is eval-harness truncation** — the 1,200-char context cap starved the answerer of lower-ranked
  moments. Full top-5 context lifts correctness **0.39 → 0.52**. (Largely an eval artifact; production
  assembles context differently — so real-product correctness ≈ 0.52, not 0.41.)
- **~24 pts is a GENUINE extraction gap** (0.52 vs 0.76 retrieval), and it survived every fix:
  - stronger answerer doesn't help — Sonnet 4.6 (0.50) ≈ Haiku 4.5 (0.54), full context, same gold set;
  - it's not a metric artifact — a Gemini *semantic* judge scores it even LOWER (0.39 vs factMatch 0.50),
    so factMatch isn't undercounting (if anything it's lenient).
  Conclusion: the answer's tokens are *present* in noisy OCR fragments but not in a cleanly answerable
  form, and "answer-bearing" (token-match on episode text) over-counts those. **The real lever is upstream
  — capture/OCR quality + retrieval precision (surface the moment that actually ANSWERS, not one that
  merely contains the tokens)** — not reranking (refuted), not a bigger answerer (refuted), not context size.

(Earlier "retrieval is the sole bottleneck, answer layer solid" was an artifact of the polluted corpus.)

## Where the levers actually are (end-of-investigation map)
1. **Capture quality** — cleaner OCR / richer structured episodes so retrieved moments are answerable, not
   just token-bearing. Biggest unexplored lever; aligns with collecting a clean *and high-fidelity* corpus.
2. **Retrieval precision** — first-stage embeddings tuned for short noisy text (exact hit@5 0.57 has room).
3. **A powered, ratified gate** on a fresh clean+high-quality capture (this external slice gave N=46, good
   enough to be directional; a purpose-built capture makes it a real gate).
NOT levers (measured-dead this round): cross-encoder reranking, answerer model size, context-window size.
